Q1 C/C++ Language
97% correct
Consider the program below. What does main return?
struct S { static int x; };
int S::x = 0;
int main(void) {
    S::x++;        // x = 1
    S s; s.x++;    // x = 2
    S t; t.x++;    // x = 3
    return S::x;
}
3.0097%
Something else3%

Answer: 3

static member variables belong to the class, not to any instance. There is exactly one copy of S::x shared by all instances.

  • S::x++ → accesses via class name, x becomes 1
  • s.x++ → accesses the same static variable through instance s, x becomes 2
  • t.x++ → same variable again through instance t, x becomes 3
Key concept: static members are per-class, not per-object. Accessing via obj.x and Class::x refers to the same memory. The static int x; inside the struct is only a declaration — the definition int S::x = 0; is required outside.
Q2 Undefined Behavior
96% correct
Given that x was declared as int x[3], accessing x[3] is undefined behaviour.
True96%
False4%

Answer: True

int x[3] allocates indices 0, 1, 2. Accessing x[3] reads or writes past the end of the array.

Array bounds in C/C++: Unlike Java or Python, C/C++ performs no bounds checking at runtime. Out-of-bounds access is undefined behavior — it might read garbage, corrupt adjacent data, or crash. This is one of the most common sources of security vulnerabilities (buffer overflow).
Subtlety: Taking the address &x[3] (pointer to one-past-the-end) is actually allowed by the standard — you just can't dereference it. This is how iterators like vec.end() work.
Q3 Undefined Behavior
95% correct
Which of the following is true for C++?
Signed overflow UB & unsigned overflow UB3%
Signed overflow defined & unsigned overflow defined0%
Signed overflow UB & unsigned overflow defined95%
Signed overflow defined & unsigned overflow UB1%

Answer: Signed overflow is UB, unsigned overflow is defined (wraps around)

Signed integer overflow is undefined behavior per the C/C++ standard. The compiler is free to assume it never happens, and may optimize accordingly.

Unsigned integer overflow is well-defined: it wraps around modulo 2N (where N is the number of bits).

Why this matters for compilers: Because signed overflow is UB, the compiler can assume x + 1 > x is always true for signed x. This enables loop optimizations. For unsigned, x + 1 could wrap to 0, so the compiler cannot make that assumption.
Real-world consequence: GCC with -O2 may remove checks like if (x + 1 < x) for signed integers, since it "can't happen" per the standard. Use -fwrapv to make signed overflow wrap, or -ftrapv to trap on it.
Q4 STL & Data Structures
99% correct
The STL contains a generic undirected graph.
True1%
False99%

Answer: False

The C++ Standard Template Library (STL) provides containers like vector, map, set, unordered_map, list, deque, stack, queue, priority_queue, etc. But there is no graph data structure.

If you need a graph in C++: You typically build one yourself using adjacency lists (vector<vector<int>>) or adjacency maps, or use a library like Boost.Graph. In SVF, graphs are implemented with custom GenericGraph<NodeType, EdgeType> template classes.
Q5 STL & Data Structures
89% correct
Which of the following has the poorer worst case time complexity?
(1) A binary search on a sorted array for a specific element.
(2) A search in a sorted binary tree for a specific item.
(1)4%
(2)89%
They have the same worst case7%

Answer: (2) — Binary tree search has worse worst case

Binary search on sorted array: Always O(log n) — the array is always balanced by definition.

Search in a (sorted) binary tree: A "sorted binary tree" (BST) can be degenerate (essentially a linked list), making worst-case search O(n).

Key distinction:
  • Sorted array + binary search → always O(log n)
  • BST → O(log n) average, O(n) worst case (skewed tree)
  • Balanced BST (AVL/Red-Black) → O(log n) guaranteed
The question says "sorted binary tree," not "balanced binary tree." A BST where elements were inserted in sorted order becomes a linked list.
Note: std::map uses a red-black tree (self-balancing), so it guarantees O(log n). A plain BST does not.
Q6 C/C++ Language
91% correct
How many times is function fib called?
unsigned int fib(unsigned int n) {
    if (n < 2) return n;
    else return fib(n - 1) + fib(n - 2);
}
int main(void) { fib(11); }
28791%
Something else9%

Answer: 287

Let C(n) = total number of calls to compute fib(n). C(0) = 1, C(1) = 1, and for n ≥ 2: C(n) = 1 + C(n-1) + C(n-2).

Computing step by step:

  • C(0) = 1, C(1) = 1
  • C(2) = 1+1+1 = 3
  • C(3) = 1+3+1 = 5
  • C(4) = 1+5+3 = 9
  • C(5) = 1+9+5 = 15
  • C(6) = 1+15+9 = 25
  • C(7) = 1+25+15 = 41
  • C(8) = 1+41+25 = 67
  • C(9) = 1+67+41 = 109
  • C(10) = 1+109+67 = 177
  • C(11) = 1+177+109 = 287
Background: The naive recursive Fibonacci has exponential time complexity O(2n). The number of calls C(n) = 2·fib(n+1) - 1. This is why memoization or iterative approaches are essential. With memoization, it becomes O(n).
Q7 Memory & Pointers
95% correct
It is possible to access memory unsafely through pointers returned by malloc but any access to memory through pointers returned by new is always safe.
True5%
False95%

Answer: False

Both malloc and new return pointers, and both can be used unsafely.

  • Use-after-free: delete p; p->x; — UB
  • Double free: delete p; delete p; — UB
  • Array overrun: int* a = new int[3]; a[10] = 5; — UB
  • Mismatched delete: int* a = new int[3]; delete a; — UB (should be delete[])
Difference between malloc and new:
  • malloc: C function, returns void*, does NOT call constructors, freed with free()
  • new: C++ operator, returns typed pointer, calls constructors, freed with delete
  • Both allocate heap memory, both can lead to memory errors if misused
Q8 Security
98% correct
In which version of C was the gets function removed?
C890%
C1198%
C231%
C991%

Answer: C11

gets() reads a line from stdin with no buffer size limit, making buffer overflows inevitable. Timeline:

  • C89/C90: gets() introduced — no size parameter
  • C99: gets() deprecated (still available)
  • C11: gets() removed from the standard, replaced by gets_s() in Annex K
Why gets() is dangerous: gets(buf) reads until newline with no bounds check. If input is longer than buf, it overwrites adjacent stack memory. The Morris Worm (1988) exploited a gets() buffer overflow in fingerd.
Safe alternative: fgets(buf, sizeof(buf), stdin) — takes a size parameter, preventing overflow.
Q9 Security

In which version of C++ was the insecure std::gets function removed?

C++173%

Answer: C++14

C++ tracks C standard library changes but on a slightly later schedule:

  • C11 removed gets() from C
  • C++11 deprecated std::gets
  • C++14 officially removed std::gets
Why the lag? C++ inherits the C standard library. C11 removed gets, and C++14 adopted that change. C++11 was released around the same time as C11 and only marked it deprecated.
Q10 Memory & Pointers
49% correct ⚠ Trickiest question!
A null pointer dereference is possible when accessing what a C++ reference refers to (e.g., T &t = ...; t;)?
True51%
False49%

Answer: False

This was the hardest question (only 49% correct — essentially a coin flip). The reasoning is nuanced:

A valid C++ reference cannot be null. By the language rules, a reference must always refer to a valid object. Therefore, using a reference is defined to be safe — you cannot get a null pointer dereference through normal reference usage.

But wait — can't you do this? 
int* p = nullptr;
int& r = *p;  // ← UB happens HERE, not when using r
r;            // this line is irrelevant — UB already occurred
The undefined behavior occurs at the point of creating the reference from a null pointer (*p). The reference itself, once it exists, is assumed to be valid. The question asks about accessing what a reference refers to, not about creating a reference from a null pointer. A properly constructed reference cannot be null.
Key rule: References in C++ are designed as "always-valid aliases." The language provides no syntax like r == nullptr for references. If a reference exists, it refers to a valid object (by contract). Any code that creates a null reference has already invoked UB before the reference is used.
Q11 Undefined Behavior
94% correct
Dereferencing a null pointer will always cause the program to crash.
True6%
False94%

Answer: False

Dereferencing a null pointer is undefined behavior. "Undefined" means anything can happen:

  • It might crash (segfault) — this is the common case on most OSes
  • It might silently read/write address 0 (e.g., embedded systems with no MMU)
  • The compiler might optimize it away entirely (since UB means the code is "unreachable")
  • It might appear to work and corrupt memory
Key distinction: Segfault is an OS/hardware protection mechanism, not a language guarantee. On systems without virtual memory (microcontrollers), address 0 may be valid, readable memory. The C/C++ standard makes no promise about crashing — it says the behavior is simply undefined.
Q12 Undefined Behavior
79% correct
What will the following program print?
#include <stdio.h>
int main(void) {
    int x;           // local variable, NOT initialized
    if (x) puts("hello");
    puts("bye");
}
hellobye7%
[nothing]3%
[unknown]79%
bye11%

Answer: [unknown]

x is a local (automatic) variable that is never initialized. Reading an uninitialized local variable is undefined behavior in C/C++.

  • It could be 0, some garbage value, or the compiler could do anything
  • The if-branch might or might not execute
  • The compiler might even optimize the entire program unpredictably
Why not "bye"? Many students assume x will be 0 (like in Java). In C/C++, local variables are not zero-initialized. The stack memory contains whatever was there before. 11% chose "bye" assuming x == 0, but that's not guaranteed.
Compare with Q13: If x were a global variable, it would be zero-initialized. That's the key difference.
Q13 Undefined Behavior
95% correct
What will the following program print?
#include <stdio.h>
int x;               // GLOBAL variable → zero-initialized
int main(void) {
    if (x) puts("hello");
    puts("bye");
}
[unknown]2%
[nothing]0%
hellobye3%
bye95%

Answer: bye

x is a global variable (file scope, static storage duration). In C/C++, global variables are zero-initialized before program startup.

So x == 0, the if-branch is skipped, and only "bye" is printed.

Storage duration rules:
  • Automatic (local): NOT initialized → reading is UB (Q12)
  • Static (global, or static local): zero-initialized before any code runs
  • Dynamic (new/malloc): malloc does NOT initialize; new int() with () zero-initializes, new int without () does NOT
Q14 STL & Data Structures
97% correct
Iterating over a std::map twice will yield the same order.
True97%
False3%

Answer: True

std::map is an ordered associative container (implemented as a red-black tree). Elements are always sorted by key. Iterating always yields the same order (ascending by key).

Contrast with std::unordered_map: An unordered_map uses a hash table. While iterating twice without modification will likely give the same order, the standard does not guarantee it. Rehashing (triggered by insertions) can change the order.
Q15 STL & Data Structures
97% correct
Dereferencing an invalidated iterator is well defined and results in a no-op.
True3%
False97%

Answer: False

Dereferencing an invalidated iterator is undefined behavior, not a no-op.

Common causes of iterator invalidation:
  • vector: any insertion that triggers reallocation invalidates ALL iterators
  • vector::erase(): invalidates iterators at and after the erased element
  • map/set::erase(): only invalidates the erased element's iterator
  • unordered_map: insertion may trigger rehash, invalidating all iterators
This is a common bug source. Erasing elements while iterating a vector: for (auto it = v.begin(); it != v.end(); ++it) { if (cond) v.erase(it); } — this is UB. Use the erase-remove idiom or it = v.erase(it).
Q16 Memory & Pointers
66% correct (multiple choice)
Which of the following may cause a stack overflow? (Select all that apply)
Accessing a stack-allocated array out of bounds11%
Dereferencing a null pointer allocated on the stack2%
Calling a recursive function without a base case97%
Allocating an array on the stack larger than physical memory74%

Answer: Recursive function without base case + Array larger than physical memory

Stack overflow = the stack pointer goes beyond the allocated stack space.

  • Infinite recursion (97% got this): Each function call pushes a new stack frame. Without a base case, the stack grows until it overflows.
  • Huge stack array (74% got this): int arr[1000000000]; tries to reserve ~4GB on the stack. The default stack size is typically 1-8 MB. This overflows the stack immediately.
Why NOT the other two?
  • Out-of-bounds array access: This is a buffer overflow (corrupts adjacent memory), not a stack overflow. These are different bugs with different names.
  • Null pointer dereference: This causes a segfault (accessing address 0), not a stack overflow. The stack is not involved.
Terminology: "Stack overflow" specifically means exhausting stack space. "Buffer overflow" means writing past the end of a buffer. They are different classes of vulnerabilities.
Q17 Tools
74% correct (multiple choice)
Which of the following tools can help diagnose memory errors? (Select all)
GDB81%
make1%
Valgrind100%
UBSan90%

Answer: GDB, Valgrind, UBSan

  • Valgrind (100% selected): Dynamic analysis tool. Its Memcheck tool detects use-after-free, uninitialized reads, memory leaks, double frees. Runs program in a virtual CPU — slow (10-50x) but very thorough.
  • UBSan (90%): Undefined Behavior Sanitizer. Compile with -fsanitize=undefined. Detects signed overflow, null dereference, out-of-bounds, etc. at runtime with minimal overhead (~2x).
  • GDB (81%): Debugger. Not specifically a memory error tool, but you can catch segfaults, inspect memory, set watchpoints on addresses. It helps diagnose memory errors after they occur.
  • make (1%): Build tool. Has nothing to do with memory errors.
Other useful tools:
  • ASan (AddressSanitizer): -fsanitize=address — detects out-of-bounds, use-after-free, leaks. Faster than Valgrind (~2x slowdown).
  • MSan (MemorySanitizer): -fsanitize=memory — detects uninitialized memory reads.
  • Static analyzers (e.g., SVF, Clang Static Analyzer): Find bugs at compile time without running the program.
Q18 Security
99% correct
Security can be ignored for software designed to run offline locally rather than on a network-connected server.
True1%
False99%

Answer: False

Offline software still processes untrusted input (files, USB drives, user data). Attack vectors include:

  • Malicious files (PDFs, images, documents) that exploit parser vulnerabilities
  • Privilege escalation from local exploits
  • Malware that spreads via removable media (Stuxnet spread via USB)
  • Data integrity — a buffer overflow in an offline app can still corrupt/steal local data
Principle: Security is about protecting against untrusted input, not just network attacks. Any software that processes external data needs to be secure.
Q19 Security
90% correct
Software foo dynamically links library bar. The developers of bar just fixed a major security vulnerability but the API remained stable. What do the users of foo need to do to ensure they are safe?
Do nothing, bar is dynamically linked4%
Download both the latest foo and bar3%
Download the latest foo3%
Download the latest bar90%

Answer: Download the latest bar

With dynamic linking, the library (bar.so / bar.dll) is loaded at runtime. Updating just the library file fixes the vulnerability for all programs that use it, without recompiling anything.

Dynamic vs. Static linking:
  • Dynamic: Library loaded at runtime. Updating the .so/.dll file is enough. The main program doesn't need to change.
  • Static: Library code is baked into the executable at compile time. To fix a vulnerability, you must recompile foo with the new bar. Every statically-linked program needs to be rebuilt.
This is a major advantage of dynamic linking for security patches.
Why not "do nothing"? Dynamic linking means the library is separate, but you still need to update it. The old, vulnerable version of bar is still on disk until replaced.
Q20 Security
96% correct
fgets is prone to buffer overflows in the same way gets is.
True4%
False96%

Answer: False

fgets takes a size parameter, which gets does not.

// gets — NO size limit → buffer overflow
char buf[64];
gets(buf);  // reads unlimited input!

// fgets — size parameter → safe
char buf[64];
fgets(buf, sizeof(buf), stdin);  // reads at most 63 chars + null
fgets behavior: Reads at most n-1 characters (reserves 1 byte for \0). Stops at newline or EOF. The newline character IS included in the buffer (unlike gets). This is the standard safe replacement for gets.
