Q1 Memory & Pointers
In the above program, which of the following are true? (Select all)
#include <assert.h>
#include <stdio.h>
#include <stdlib.h>

int main() {
    int *p = (int *)malloc(sizeof(int)); assert(p != NULL);
    int *q = (int *)malloc(sizeof(int)); assert(q != NULL);
    *p = 10;                          // line 8
    *q = 20;                          // line 9
    p = q;                            // line 10: p now points to q's memory
    printf("*p is %d\n", *p);       // line 11
    free(p);                          // line 12: frees q's memory
    printf("*q is %d\n", *q);       // line 13
    free(q);                          // line 14
    return 0;
}
The program triggers a use-after-free at line 10Incorrect
The program triggers a use-after-free at line 13Correct
The program triggers a null pointer dereference at line 13Incorrect
The program triggers a double-free at line 14Correct

Answer: Use-after-free at line 13 + Double-free at line 14

Let's trace the pointer state step by step:

  • Line 6-7: p → block A, q → block B
  • Line 10: p = q → now both p and q point to block B. Block A is leaked (no pointer to it anymore).
  • Line 12: free(p) → frees block B (since p now points to B).
  • Line 13: *q → q still points to block B, which was just freed. This is a use-after-free.
  • Line 14: free(q) → tries to free block B again. This is a double-free.
Why NOT "use-after-free at line 10"? Line 10 is just a pointer assignment (p = q). No dereference occurs — it just copies the address. No memory is accessed through either pointer on that line.
Memory leak: This program also leaks block A. After p = q, there's no way to reach or free block A. A complete bug list: memory leak (block A), use-after-free (line 13), double-free (line 14).
Q2 Integer Overflow
The above program may trigger a signed integer overflow at the line marked with X.
#include <stdlib.h>
#include <stdio.h>

int main(int argc, char *argv[]) {
    if (argc < 3) return 1;
    long n = strtol(argv[1], NULL, 10);
    long m = strtol(argv[2], NULL, 10);
    printf("%ld + %ld = %ld\n", n, m, n + m);  // X
    return 0;
}
TrueCorrect
FalseIncorrect

Answer: True

strtol returns a long value. The user can provide any values via command line, including LONG_MAX. If both n and m are large positive (or large negative) values, n + m can overflow.

long is a signed type, so overflow is undefined behavior.

Key rule: Signed integer overflow is UB in C/C++. The compiler may assume it never happens and optimize accordingly. strtol itself clamps to LONG_MIN/LONG_MAX on overflow (and sets errno), but the addition n + m has no such protection.
Safe pattern: To safely add two longs, check before the operation: 
if (m > 0 && n > LONG_MAX - m) { /* overflow! */ }
if (m < 0 && n < LONG_MIN - m) { /* underflow! */ }
Q3 Memory & Pointers
The above program may trigger a null pointer dereference at the line marked with X.
#include <stdlib.h>
#include <stdio.h>

int main(int argc, char *argv[]) {
    int *p = (int *)malloc(sizeof(int));
    *p = argc;  // X
    printf("argc is %d\n", *p);
    return 0;
}
TrueCorrect
FalseIncorrect

Answer: True

malloc can return NULL if memory allocation fails (out of memory). The code does not check the return value before dereferencing p.

If malloc returns NULL, then *p = argc dereferences a null pointer — undefined behavior.

Always check malloc's return value: 
int *p = (int *)malloc(sizeof(int));
if (p == NULL) {
    perror("malloc failed");
    return 1;
}
*p = argc;  // now safe
Compare with Q1: that code uses assert(p != NULL) to guard against this. This program does not.
Note: On most modern systems with overcommit (Linux), malloc almost never returns NULL for small allocations. But relying on this is non-portable and dangerous.
Q4 Memory & Pointers
Assuming the rest of our list functions are implemented correctly, our list implementation contains a memory leak (i.e., looking at just free_list, ensure all allocated structs would be freed).
typedef struct node {
    int data;
    struct node *next;
} node;

typedef struct list {
    size_t num_nodes;
    node *head;
} list;

void free_list(list *l) {
    if (l == NULL) return;
    if (l->head == NULL) return;

    node *current = l->head;
    while (current != NULL) {
        node *next = current->next;
        free(current);
        current = next;
    }
}
TrueCorrect
FalseIncorrect

Answer: True — the list struct itself is never freed

The function correctly frees all nodes in the linked list. However, it never frees the list struct l itself.

If l was allocated with malloc, it leaks. The fix:

void free_list(list *l) {
    if (l == NULL) return;
    node *current = l->head;
    while (current != NULL) {
        node *next = current->next;
        free(current);
        current = next;
    }
    free(l);  // don't forget the list struct!
}
Common pattern: When freeing a composite data structure, you must free from the inside out: first the elements (nodes), then the container (list). Missing either level causes a leak.
Q5 Security & Strings
buf is a 50-byte buffer on the stack. Using fgets, we want the user to populate buf with some data. Which calls to fgets will avoid a buffer overflow and allow the user to fill the entirety of buf?
fgets(buf, 49, stdin)Incorrect
fgets(buf, (sizeof(buf) - 1) * sizeof(char), stdin)Incorrect
fgets(buf, 50 * sizeof(char), stdin)Correct
fgets(buf, sizeof(buf) * sizeof(char), stdin)Correct

Answer: fgets(buf, 50, stdin) and fgets(buf, sizeof(buf), stdin)

fgets(buf, n, stdin) reads at most n-1 characters and appends a null terminator. So the second argument should be the full buffer size, not size-1.

  • fgets(buf, 49, stdin) — reads at most 48 chars. Wastes 1 byte of the buffer. Incorrect (doesn't fill entirety).
  • fgets(buf, (sizeof(buf)-1) * sizeof(char), stdin) — same problem, reads at most 48 chars.
  • fgets(buf, 50 * sizeof(char), stdin) — reads at most 49 chars + null. Correct. (sizeof(char) is always 1).
  • fgets(buf, sizeof(buf) * sizeof(char), stdin) — same as 50 * 1 = 50. Correct.
fgets behavior: fgets(buf, n, stream) reads at most n-1 characters, then writes \0. The n parameter is the total buffer size including the null terminator. Pass sizeof(buf), not sizeof(buf)-1.
Common mistake: People often subtract 1 "to leave room for the null terminator" — but fgets already accounts for that. Subtracting 1 wastes a byte.
Q6 Memory & Pointers
Which of the following statements is true?
#include <stdlib.h>
#include <stdio.h>
#include <assert.h>

int main() {
    int *data = (int *)malloc(100 * sizeof(int));
    assert(data != NULL);

    for(int i = 0; i < 100; i++) {     // line 8
        data[i] = 5;
    }

    free(data);                         // line 10
    printf("%d\n", data[0]);           // line 11
    return 0;
}
A double-free occurs at line 10Incorrect
A buffer overflow occurs at line 8Incorrect
A use-after-free occurs at line 11Correct
A null pointer dereference may occur at line 8Incorrect

Answer: Use-after-free at line 11

  • Line 10: free(data) releases the allocated memory. The pointer data becomes a dangling pointer.
  • Line 11: data[0] accesses the freed memory → use-after-free (undefined behavior).

Why NOT the other choices:

  • No double-free: free is only called once.
  • No buffer overflow: loop goes from 0 to 99, and we allocated 100 ints. Exactly right.
  • No null dereference: the assert guards against that.
Best practice: After free(p), set p = NULL to avoid accidental use-after-free. A null dereference crashes immediately (easier to debug) rather than silently corrupting data.
Q7 Security & Strings
buf is a buffer of 50 ints. Which of the following calls to memset will result in a buffer overflow?
memset(buf, 0, 50)No overflow
memset(buf, sizeof(int) * sizeof(buf), 100)No overflow
memset(buf, 0, 0)No overflow
memset(buf, 100, sizeof(int) * sizeof(buf) * 50)Overflow!

Answer: memset(buf, 100, sizeof(int) * sizeof(buf) * 50)

memset(ptr, value, num_bytes) — the third argument is the number of bytes to set.

For int buf[50]: sizeof(buf) = 50 × 4 = 200 bytes (assuming 4-byte int).

  • memset(buf, 0, 50) — sets 50 bytes. Only covers 12.5 ints. No overflow (writes less than buffer size).
  • memset(buf, ..., 100) — sets 100 bytes. Only 25 ints worth. No overflow.
  • memset(buf, 0, 0) — sets 0 bytes. No-op. No overflow.
  • memset(buf, 100, sizeof(int) * sizeof(buf) * 50) — sets 4 × 200 × 50 = 40,000 bytes. The buffer is only 200 bytes. Massive overflow!
Note on argument 2: memset's second argument is the value (as unsigned char), not a count. memset(buf, sizeof(int)*sizeof(buf), 100) sets 100 bytes to the value 800 (truncated to 0x20 = 32). Confusing but not a buffer overflow.
Correct usage: memset(buf, 0, sizeof(buf)) — zeroes out the entire buffer safely. Always use sizeof for the third argument.
Q8 Assertions & Tools
Inserting assertions never affects performance.
TrueIncorrect
FalseCorrect

Answer: False — assertions DO affect performance

Assertions execute a condition check at runtime. If the condition involves computation (function calls, complex expressions), it adds overhead.

  • An assertion inside a hot loop can significantly slow down the program.
  • In release builds, assertions are typically disabled via #define NDEBUG (which makes assert() expand to nothing).
  • But in debug builds, every assertion is evaluated at runtime.
The NDEBUG mechanism: 
// With NDEBUG defined, assert() becomes a no-op:
#define NDEBUG
#include <assert.h>
// assert(expr) expands to ((void)0)

// Without NDEBUG, assert() evaluates the expression:
// assert(expr) → if (!(expr)) abort()
Compile with -DNDEBUG for release builds to remove assertion overhead.
Q9 Undefined Behavior
The following program may trigger undefined behaviour.
#include <stdlib.h>
#include <stdio.h>

void tenacc(void) {
    static unsigned int acc = 0;
    acc += 10;
    printf("Accumulator at %u\n", acc);
}

int main(int argc, char *argv[]) {
    if (argc < 2) return 1;
    long n = strtol(argv[1], NULL, 10);
    for (long i = 0; i < n; ++i) tenacc();
    return 0;
}
TrueIncorrect
FalseCorrect

Answer: False — no UB possible

The key insight: acc is unsigned int.

  • Unsigned integer overflow is well-defined in C/C++. It wraps around modulo 232 (for 32-bit unsigned int).
  • Even if the user provides a huge n, acc += 10 will just keep wrapping. No UB.
  • The %u format specifier correctly matches unsigned int.
  • strtol returns a long, and the loop variable i is long — no issues there.
Recall from Quiz 1 Q3: Signed integer overflow is UB, but unsigned integer overflow is defined (wraps modulo 2N). This question tests whether you can distinguish between the two. If acc were int (signed), this WOULD be UB.
Q10 Security & Strings
password is a global buffer containing a user inputted password (null terminated string). The correct password is "hello". The check_password function returns non-0 only when the user entered the correct password and 0 otherwise.
#include <string.h>

int check_password(void) {
    return strncmp(password, "hello", 5) == 0;
}
TrueIncorrect
FalseCorrect

Answer: False — the check is bypassable

strncmp(password, "hello", 5) only compares the first 5 characters. Any string that starts with "hello" will pass:

  • "hello" → passes (correct)
  • "helloworld" → also passes! (incorrect — this is not the right password)
  • "hello123" → also passes!

The fix: use strcmp (compares entire strings) or compare with length 6 to include the null terminator:

// Option 1: use strcmp
return strcmp(password, "hello") == 0;

// Option 2: strncmp with length including '\0'
return strncmp(password, "hello", 6) == 0;
Security lesson: String prefix matching is a classic authentication bypass. Always compare the full string including the null terminator when checking passwords or tokens.
Q11 Undefined Behavior
The atoi function can trigger undefined behaviour.
TrueCorrect
FalseIncorrect

Answer: True

atoi converts a string to an int. If the result is outside the range of int, the behavior is undefined per the C standard (C11 §7.22.1).

  • atoi("99999999999") — value exceeds INT_MAX → UB
  • atoi has no way to report errors (no errno, no return code for failure)
  • atoi("") → returns 0 (defined, not UB)
  • atoi(NULL) → UB (null pointer dereference)
Safe alternative: strtol 
long val = strtol(str, &endptr, 10);
if (errno == ERANGE) { /* overflow */ }
if (endptr == str) { /* no digits found */ }
strtol sets errno on overflow and provides error checking via endptr. Always prefer strtol over atoi.
Q12 Undefined Behavior
There is a bug in the above code, what is it?
struct collection {
    int a;
    int b;
};

int main(void) {
    struct collection *c = (struct collection *)malloc(sizeof(struct collection));
    assert(c != NULL);
    printf("Fresh: a = %d, b = %d\n", c->a, c->b);  // BUG
    c->a = 10; c->b = 20;
    printf("Updated: a = %d, b = %d\n", c->a, c->b);
    return 0;
}
We potentially perform null pointer dereferencesIncorrect
We read undefined variablesCorrect
We allocate the incorrect number of bytes in the malloc callIncorrect
We allocate c on the heap even though we could have allocated it on the stackIncorrect

Answer: We read undefined (uninitialized) variables

malloc allocates memory but does not initialize it. After malloc, c->a and c->b contain whatever garbage was in that memory.

The first printf reads c->a and c->b before they are assigned — this reads indeterminate values (UB in C).

malloc vs calloc:
  • malloc(size) — does NOT initialize. Memory contains garbage.
  • calloc(count, size) — zero-initializes all bytes to 0.
If you need zero-initialized memory, use calloc, or explicitly set fields after malloc.
Why not "null pointer dereference"? The code has assert(c != NULL), which guards against that. The assert ensures c is valid before use.
Q13 Security & Strings
There is a bug in the above program, what is it?
int main(void) {
    char password[4];
    password[0] = 'p';
    password[1] = 'a';
    password[2] = 's';
    password[3] = 's';
    char input[10] = { '\0' };
    fgets(input, 10, stdin);
    return strcmp(password, input);
}
A buffer overflow may occur when we are inserting "pass" into passwordIncorrect
A buffer overflow may occur through fgetsIncorrect
A buffer overflow may occur through strcmpCorrect
A signed integer overflow may occur at the return statementIncorrect

Answer: Buffer overflow through strcmp

password is {'p','a','s','s'} — no null terminator! The array is exactly 4 bytes with no room for '\0'.

strcmp scans both strings until it finds a '\0'. Since password has no null terminator, strcmp will read past the end of the array, looking for a '\0' that may never come — buffer over-read.

Why NOT the other options?
  • "pass" into password: We manually set indices 0-3, which is exactly within bounds of char[4]. No overflow.
  • fgets: fgets(input, 10, stdin) reads at most 9 chars into a 10-byte buffer. Safe.
  • strcmp returns an int. return just passes that value. No integer overflow concern.
Lesson: C strings MUST be null-terminated. A char array without '\0' is not a valid C string. Any string function (strcmp, strlen, strcpy, printf %s) will read past the end looking for the terminator.
Q14 C/C++ Language
When allocating memory for ints, it is safe to assume an int is 4 bytes (e.g., int *p = (int *)malloc(4)).
TrueIncorrect
FalseCorrect

Answer: False — never hardcode type sizes

The C standard only guarantees minimum sizes:

  • char: at least 8 bits (always exactly 1 byte by definition)
  • short: at least 16 bits
  • int: at least 16 bits
  • long: at least 32 bits

While int is 4 bytes on most modern platforms (x86, ARM), it can be 2 bytes on some embedded systems, or potentially different sizes on exotic architectures.

Always use sizeof: 
// Wrong — assumes int is 4 bytes
int *p = (int *)malloc(4);

// Correct — portable
int *p = (int *)malloc(sizeof(int));

// Even better — type-safe
int *p = (int *)malloc(sizeof(*p));
The last form (sizeof(*p)) is preferred because if you change the type of p, the allocation automatically matches.
Q15 Memory & Pointers
In this code, assuming N < 50, when does a buffer overflow occur?
int main(void) {
    int nums[N];
    for (unsigned int i = 0; i < N; i += 2) {
        nums[i] = 1;
        nums[i+1] = 0;    // potential overflow when i = N-1
    }
}
There is never a buffer overflowIncorrect
There is a buffer overflow regardless of N's valueIncorrect
There is a buffer overflow when N is evenIncorrect
There is a buffer overflow when N is oddCorrect

Answer: Buffer overflow when N is odd

The loop increments i by 2. On each iteration, it accesses nums[i] and nums[i+1].

  • N is even (e.g., N=6): i takes values 0, 2, 4. Accesses indices 0,1, 2,3, 4,5 — all within bounds. Safe.
  • N is odd (e.g., N=5): i takes values 0, 2, 4. When i=4, nums[4] is OK, but nums[4+1] = nums[5] is out of bounds (array only has indices 0-4).
Pattern to recognize: When a loop steps by 2 and accesses [i] and [i+1], the array size must be even to avoid overflow. For odd-sized arrays, the last iteration's [i+1] goes past the end.
Q16 Integer Overflow
What should CONDITION be replaced with to ensure add10 does not cause an integer overflow whilst ensuring it can handle as many values of x as possible?
#include <limits.h>
#include <assert.h>

unsigned int add10(unsigned int x) {
    assert(CONDITION);
    return x + 10;
}
UINT_MAX - 10 >= xCorrect
INT_MAX - 10 == xIncorrect
x + 10 <= UINT_MAXIncorrect
x + 10 > UINT_MAXIncorrect

Answer: UINT_MAX - 10 >= x

To check if x + 10 would overflow, you must check before the addition (checking after is too late — the overflow already happened).

  • UINT_MAX - 10 >= x — rearranges x + 10 <= UINT_MAX without performing the potentially overflowing addition. Correct.
  • INT_MAX - 10 == x — wrong constant (INT_MAX instead of UINT_MAX), and == is too restrictive (should be >=).
  • x + 10 <= UINT_MAX — this performs the addition first. If it wraps (unsigned), the result is always ≤ UINT_MAX by definition. The check is useless.
  • x + 10 > UINT_MAX — same problem; unsigned arithmetic wraps, so this is always false.
Safe overflow check pattern: To check if a + b overflows for unsigned types: 
if (a > UINT_MAX - b) { /* would overflow */ }
Always rearrange the inequality to avoid performing the potentially overflowing operation.
Q17 Integer Overflow
What should CONDITION be replaced with to ensure square does not cause an integer overflow whilst ensuring it can handle as many values of x as possible?
#include <limits.h>
#include <assert.h>

int square(int x) {
    assert(CONDITION);
    return x * x;
}
(x >= 0 && (INT_MAX / x) >= x) || (x < 0 && (INT_MAX / x) >= x)Correct
INT_MAX - x * x >= 0Incorrect
x * x <= INT_MAXIncorrect
(x >= 0 && (INT_MAX / x) >= x) || (x < 0 && (INT_MIN / x) >= x)Incorrect

Answer: Check both positive and negative cases against INT_MAX

Since x * x is always non-negative (for real results), we only need to check against INT_MAX, not INT_MIN.

  • If x >= 0: check INT_MAX / x >= x (i.e., x² ≤ INT_MAX)
  • If x < 0: x * x is the same as |x| * |x|. We check INT_MAX / x >= x. Since both are negative, dividing two negatives gives a positive, and the comparison works correctly.
Why NOT x * x <= INT_MAX? This computes x * x first — if it overflows, we already have UB. The check must be done before the multiplication.
Why NOT the INT_MIN variant? x * x can never produce a negative result (mathematically). So checking against INT_MIN for the negative case is wrong — the potential overflow is always towards INT_MAX.
Edge case: When x == 0, INT_MAX / 0 is division by zero (UB). The condition x >= 0 && (INT_MAX / x) >= x short-circuits: when x == 0, x * x == 0 which is fine. You'd need to handle x == 0 specially for a truly robust check.
Q18 Assertions & Tools
Which of the following implementations of xmalloc (wraps malloc, aborts when malloc fails) are correct?
void *xmalloc(size_t size) { void *p = malloc(size); assert(p != NULL); return p; }Correct
void *xmalloc(size_t size) { void *p = malloc(size); assert(!errno); return p; }Incorrect
void *xmalloc(size_t size) { void *p = malloc(size); assert(errno != ENOMEM); return p; }Incorrect
void *xmalloc(size_t size) { void *p = malloc(size); if (p == NULL) abort(); return p; }Correct

Answer: assert(p != NULL) and if (p == NULL) abort()

  • assert(p != NULL): Aborts if p is NULL. Works in debug builds. Note: disabled when NDEBUG is defined! For a real xmalloc, this may not be ideal for release, but it fulfils the stated requirement.
  • if (p == NULL) abort(): Always aborts on failure, even in release builds. The most robust approach.
Why NOT errno-based checks?
  • malloc is NOT required to set errno by the C standard. Some implementations do (e.g., glibc sets errno = ENOMEM), but this is implementation-defined, not portable.
  • errno might already be non-zero from a previous call. !errno could false-trigger even if malloc succeeded.
  • The only reliable way to check malloc failure is p == NULL.
Real-world xmalloc: Libraries like Git use xmalloc that calls die("out of memory") on failure. This is simpler than checking every malloc call individually.
Q19 Assertions & Tools
The assertion in the following program ensures the program is run with command line arguments from an interactive shell. Assume the executing shell is a Strictly Conforming POSIX Application.
int main(int argc, char *argv[]) {
    assert(argc > 1);
    // ...
    return 0;
}
TrueCorrect
FalseIncorrect

Answer: True

In POSIX, when a program is run from a conforming shell, argc is always at least 1 — argv[0] is the program name. assert(argc > 1) checks that at least one command-line argument was provided (beyond the program name).

So the assertion ensures the program was run with at least one argument, e.g., ./program arg1.

argc/argv guarantee:
  • POSIX shells always set argv[0] to the program name → argc >= 1
  • argc > 1 means at least one additional argument was passed
  • argv[argc] is always NULL (guaranteed by the standard)
Subtlety: The C standard technically allows argc == 0 (e.g., when calling execve with an empty argv array). But in a "Strictly Conforming POSIX Application" shell, this doesn't happen.
Q20 C/C++ Language
Deleting an element from a std::set invalidates all current iterators into that std::set.
TrueIncorrect
FalseCorrect

Answer: False — only the erased element's iterator is invalidated

std::set is implemented as a red-black tree (balanced BST). Erasing a node only affects that specific node. All other iterators remain valid.

Iterator invalidation rules by container:
ContainerOn eraseOn insert
vectorErased + all afterAll (if reallocation)
dequeAllAll
listOnly erasedNone
set/mapOnly erasedNone
unordered_set/mapOnly erasedAll (if rehash)
Node-based containers (set, map, list) have the best iterator stability because each element is an independent node in memory.
